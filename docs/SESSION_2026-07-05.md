## 1. Goal

Make T3MP3ST **local-model-friendly** and replace the **hardcoded model list**
with **standardized live discovery**, in the original author's style, drivable
headlessly (Docker), and documented for delivery from the fork (no upstream PR).
By the end: the app must **actually route** LLM traffic to the selected provider —
including a local OpenAI-compatible server.

## 2. What we delivered

### Backend / engine
- **`custom` provider** — local/self-hosted OpenAI-compatible server; key optional
  (keyless supported). Reuses the OpenAI adapter (`CustomAdapter`).
- **`nvidia` provider** — NIM at `integrate.api.nvidia.com/v1` (`NvidiaAdapter`).
- **`src/llm/discovery.ts`** — live `/models` (OpenAI shape) + `/v1/models`
  (Anthropic shape), non-throwing, per-wire-shape. Cached 24h, keyed by baseUrl.
- Config: `custom`/`nvidia` wired into `getLLMConfig`, `getConfiguredProviders`,
  fallback chain, setters; new `setBaseUrl`, `getModels`, `refreshModels`,
  `modelCache`; `LLM_PROVIDER`/`LLM_MODEL`/`CUSTOM_*` env drivability.
- `exportConfig` redaction made generic (was omitting venice/nvidia/custom).

### CLI / setup
- Editable base URL, dynamic model picker (live + refresh + manual + fallback
  notice), `validate` / `refresh-models` / `models --refresh` commands.

### Web dashboard (`docs/index.html`, `/ui`)
- **NVIDIA** + **Custom / Local Server** settings sections (base URL + optional key).
- **Live model selector** (provider dropdown, ↻ Refresh, status line, manual entry)
  backed by the new **`POST /api/models`** server endpoint.

### Docker / security
- New `Dockerfile` + `scripts/docker-entrypoint.sh`: **no secret in the image**,
  mock placeholders, **runtime key injection**, non-blocking on-start validate.

### The routing fix (the headline)
- The provider selection was cosmetic until this — five defects (server dropped
  `baseUrl`; operators never got it; endpoints ignored per-request provider;
  client was openrouter-only; chat hardcoded `openrouter.ai`). All fixed.
  Full detail in `ROUTING_FIX.md`.

## 3. Findings (the expensive lessons)

1. **Two config surfaces.** Core (`src/config` + `.env` + `conf` store) vs the
   **client-side dashboard** (`docs/index.html`, localStorage, its own model
   list). Backend/CLI work did **not** surface in the UI until the UI was edited
   separately. Don't assume "settings" means one place.
2. **Settings ≠ routing.** A provider can be configured and its models discovered
   while requests still go elsewhere. The request path (client → server → adapter)
   is a separate surface that must be wired independently. Verify with a real
   completion, not a model list.
3. **`resolveGeneralLLMConfig` silently dropped `baseUrl`** — the single most
   important line; without it `custom` can never run.
4. **Client hardcoded OpenRouter** in two ways: `getApiKey()` (openrouter key
   only) and `_safeLLMCallOnce` (`openrouter.ai` URL). Both had to move to a
   selected-provider model (`llmRequestConfig()`), and chat had to route through
   the server (browser can't reach a local server — CORS).
5. **`.local` mDNS doesn't resolve in containers**; and **`192.0.2.99` is a
   dead IP** (real host `your-server.local` = `192.0.2.10`). Most "fetch failed"
   reports were the wrong IP, not code. Mitigated with `--add-host`.
6. **Baked secret** in the original Dockerfile (`sk-…`) — removed; runtime
   injection only.
7. **`ENV PORT` is unused** — server reads `T3MP3ST_PORT`.

## 4. Decisions (confirmed with the user)

- New `custom` provider (not overloading `openai`).
- Static `AVAILABLE_MODELS` kept as **fallback only**, flagged "not recommended".
- Inline validate→fetch **plus** an explicit refresh action.
- Keep the four upstream providers preconfigured; **add** NVIDIA.
- Dockerfile: **zero real secrets**, mock placeholders, runtime injection.
- UI discovery via a **server-side endpoint** (Option 1), full scope (providers +
  custom + live models).
